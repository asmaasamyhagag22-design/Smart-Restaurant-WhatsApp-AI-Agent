§ 01
Executive Summary

One thread, every interaction.

Customers don't want apps. They have WhatsApp open already. This system collapses the menu, the order, the reservation, the receipt, and the support agent into a single threaded conversation — without losing the safety, reliability, or auditability of a real backend.

The Smart Restaurant WhatsApp AI Agent is a multi-tenant SaaS that lets any restaurant brand bring their full ordering experience inside WhatsApp. Customers chat in their own words — text, voice notes, photos of the menu — and the agent understands intent, retrieves the right dishes, builds the cart, confirms preferences (spice, allergies, sides), books tables, takes payment via tokenized links, and tracks the order to the door.

Under the hood it runs a LangGraph state machine driving Claude 3.5 Sonnet with strict tool-use. A Pinecone-backed RAG layer keeps menus searchable in Arabic and English. Redis holds conversation state. Postgres is the system of record. Everything is wrapped in guardrails — PII scrubbing before any LLM call, hallucination checks on prices, allergy verification, and a one-tap escalation path to a human operator.

The result is an agent that doesn't just answer — it completes work.

§ 02
Problem · Approach

Why WhatsApp, and why an agent.

Three real constraints from the field, and the design choices that fell out of them.

▲ The Friction

In the MENA region, customers abandon native restaurant apps at a 71% rate within the first week of install. Web ordering forms convert at 8–14%. The same customer, in the same minute, will reliably reply to a WhatsApp message in under 90 seconds. Restaurant operations teams were already handling DMs manually — but inconsistently, with no audit trail, and zero scale.

▲ The Constraint

A naive chatbot fails here for three compounding reasons: (1) menus are large and change daily, (2) Arabic + Arabizi + English code-switch within the same sentence, and (3) any pricing or allergy mistake is unacceptable. So we needed an agent that could reason about tools, not just generate text.

▲ The Bet

We bet that a tool-using LLM, given the right scaffolding — typed schemas, retrieval over an embedded menu, deterministic state — could outperform both a rigid IVR-style bot and a human-only operation, at a fraction of the per-order cost.

PRINCIPLE · 01

Conversation as workflow

Every turn is a state transition with a side effect on the order. The LLM never makes its own state — it proposes; the graph commits.


PRINCIPLE · 02

Retrieve, don't recite

The menu lives in Pinecone. The LLM never quotes a price or item from memory — it queries, then composes.


PRINCIPLE · 03

Guardrail twice

Pre-LLM: PII scrubbing, prompt-injection filter. Post-LLM: price reconciliation, allergy check, schema validation. Two passes, no exceptions.


PRINCIPLE · 04

Human in the loop

The agent can hand off mid-conversation. The operator picks up the full thread and the agent stays silent until released.

§ 06
AI Architecture

How the agent thinks.

The agent is not a chatbot. It's a finite-state machine where the LLM is one node among many. The graph commits state; the model only proposes.

6.1 The LangGraph state machine

Every conversation is modeled as a typed state object that flows through five named nodes. Each node is a pure function from State → State; side effects (DB writes, payment links, WhatsApp sends) only happen inside designated tool nodes, never inside the model itself.

intent · classify what the user wants

A small LLM call categorizes the turn: order, browse, reserve, track, support, smalltalk, escalate. Returns a typed enum, not free text.

claude-3-5-haiku structured output ~120 tokens
plan · pick the next tool(s)

Given intent + conversation history + current cart, the planner returns an ordered list of tools to call. The graph executes them, not the LLM.

claude-3-5-sonnet tool selection
tools · run them, deterministically

Tools are pure functions with typed input / output. The graph runs them in parallel where possible, sequenced where dependencies exist.

Pydantic schemas parallel execution
reflect · did this make sense?

Post-tool, a critic checks: is the cart total mathematically correct? Are all referenced dishes in stock? Did we accidentally promise something we can't deliver? On failure, we loop back to plan.

price reconciliation availability check
respond · compose the WhatsApp message

Renders one of: a templated card, an interactive list, a button reply, or natural prose. Output is validated against the WhatsApp Cloud schema before send.

schema validation RTL-aware

6.2 Menu RAG — retrieval is everything

The single biggest accuracy lever was forbidding the LLM from ever citing a price or item from its own knowledge. The menu is embedded at ingest time into a Pinecone index, with item names + descriptions + tags + modifiers all packed into a single embedding, both Arabic and English variants stored side by side.

Queries from the agent run with hybrid retrieval: vector similarity for fuzzy intent ("something spicy that isn't too heavy") plus structured filters ("< 150 EGP", "halal", "no nuts"). Top-K results are re-ranked by a small cross-encoder before being handed to the LLM, which then composes the actual reply.

pythonapp/tools/menu_rag.py
from pydantic import BaseModel, Field
from typing import Literal

class MenuQuery(BaseModel):
    query: str = Field(..., description="what the user is looking for")
    diet: list[Literal["halal", "vegan", "vegetarian"]] = []
    exclude_allergens: list[str] = []
    max_price: float | None = None
    spice_max: int = Field(3, ge=0, le=5)
    k: int = 5

class MenuMatch(BaseModel):
    item_id: str
    name_ar: str
    name_en: str
    price: float
    score: float
    in_stock: bool

async def query_menu(q: MenuQuery, tenant_id: str) -> list[MenuMatch]:
    # 1. embed the query (multilingual — handles AR/EN/Arabizi)
    vec = await embedder.embed(q.query)

    # 2. hybrid retrieve from Pinecone with tenant + filter constraints
    raw = await pinecone.query(
        vector=vec,
        top_k=q.k * 3,
        namespace=tenant_id,
        filter={
            "in_stock": True,
            "diet": {"$in": q.diet} if q.diet else {"$exists": True},
            "allergens": {"$nin": q.exclude_allergens},
            "price": {"$lte": q.max_price} if q.max_price else {"$exists": True},
            "spice": {"$lte": q.spice_max},
        },
    )

    # 3. cross-encoder re-rank — semantic + lexical overlap
    reranked = await reranker.rerank(q.query, raw.matches, top_k=q.k)

    # 4. validate against live inventory before returning
    return [
        MenuMatch(**m.metadata, score=m.score)
        for m in reranked
        if await inventory.is_available(m.id, tenant_id)
    ]

6.3 Prompt engineering

The system prompt is composed at request time from four sources: the global persona, the tenant-specific brand voice, the user's history, and any active campaign. We hold it to ~600 tokens — longer prompts measurably degraded tool-arg quality in our evals.

6.4 LangGraph orchestrator — actual code

pythonapp/agent/graph.py
from langgraph.graph import StateGraph, END
from .state import ConvState
from .nodes import intent_node, plan_node, tools_node, reflect_node, respond_node

def build_agent() -> StateGraph:
    g = StateGraph(ConvState)

    g.add_node("intent",   intent_node)
    g.add_node("plan",     plan_node)
    g.add_node("tools",    tools_node)
    g.add_node("reflect",  reflect_node)
    g.add_node("respond",  respond_node)

    g.set_entry_point("intent")
    g.add_edge("intent",  "plan")
    g.add_edge("plan",    "tools")
    g.add_edge("tools",   "reflect")

    # reflect either replans (loop) or responds (terminate)
    g.add_conditional_edges(
        "reflect",
        lambda s: "plan" if s.needs_replan and s.hops < 8 else "respond",
    )
    g.add_edge("respond", END)

    return g.compile(checkpointer=redis_checkpointer)

▲ Inbound webhook handler

pythonapp/api/webhooks.py
from fastapi import APIRouter, Request, HTTPException, BackgroundTasks
from .security import verify_meta_signature
from .pipeline import process_inbound

router = APIRouter(prefix="/webhook")

@router.post("/whatsapp")
async def whatsapp_inbound(req: Request, bg: BackgroundTasks):
    raw = await req.body()
    sig = req.headers.get("X-Hub-Signature-256", "")

    if not verify_meta_signature(raw, sig):
        raise HTTPException(401, "invalid signature")

    payload = json.loads(raw)

    # ACK Meta within 5s — process the message in background
    bg.add_task(process_inbound, payload)
    return {"status": "ok"}

Security · Privacy

The things not to get wrong.

Customer trust is the product. Any breach of price, allergen, or payment integrity kills the business overnight.

PII never reaches the LLM

Before any LLM call, a regex + spaCy NER pass redacts: card numbers, CVVs, national IDs, full home addresses, and any string matching [+\d]{10,} not in the allow-list. The redacted tokens are substituted back only at render time.

regex + spaCy NER · pre-hook

Zero PCI scope

We never see card data. Payment is always a one-time tokenized link from the PSP. Card details, CVV, 3DS challenge — all live on the PSP's domain. Our system only sees paid / failed / refunded.

Paymob · Stripe · Fawry

Prompt-injection defense

User text is wrapped in delimited blocks. The system prompt instructs the LLM to treat anything between <user> tags as data, not instructions. Combined with strict tool schemas, injection attempts produce no observable behavior change.

delimited input · schema-only output

Tenant isolation

Row-level security policies on every customer-data table. Pinecone namespaces are tenant-scoped. Redis keys are tenant-prefixed. A test runs nightly that attempts cross-tenant reads and alerts on any 200.

Postgres RLS · namespacing · canary

Audit log, append-only

Every tool call, every state transition, every operator takeover is written to a partitioned audit table — including the full LLM input and output. Retained 90 days hot, 2 years cold.

append-only · S3 archive

Encryption at rest + in transit

TLS 1.3 everywhere. AES-256 at rest. PSP credentials in tenants.psp_credentials are envelope-encrypted with a per-tenant KMS key.

KMS · TLS 1.3 · per-tenant keys

▲ Scale levers

Stateless app pods

FastAPI workers hold zero session state. All state lives in Redis (hot) or Postgres (durable). Pods scale on RPS independently of the agent loop.

HPA on RPS + p95 latency

Worker pool for LLM

LLM-bound work is offloaded to a separate Celery worker pool. Saturation on the LLM provider doesn't block ingress.

circuit breaker · bulkhead

Pinecone pod scaling

Per-tenant namespaces; pod count scaled per traffic class. Cold tenants share a pod, hot tenants get dedicated.

tiered allocation

Aggressive caching

Menu embeddings cached in Redis for 5 min. Intent results cached on identical-prefix turns. Cuts ~28% of LLM calls in steady state.

prefix-cache key

Prompt compression

Conversation history is summarized after turn 6 — old turns become a compressed "story so far" block. Keeps token cost linear, not quadratic.

recursive summarization

Speculative tool dispatch

When the planner is >90% confident, the most likely tool runs in parallel with the LLM's plan output. Saves ~400ms in the common case.

probabilistic execution
§ 12
Deployment

How it runs.

Three environments, one pipeline, zero hand-edits in production.

The whole platform lives in a single AWS account, region eu-central-1 for GDPR posture and MENA latency. Kubernetes (EKS) hosts the FastAPI gateway, the Celery worker pool, and the agent runtime as three separate deployments — each with its own HPA and PodDisruptionBudget. ArgoCD reconciles to whatever's in main; nothing is ever applied by hand.

Per-tenant cost is tracked through the Langfuse trace metadata. Every conversation gets a synthetic tag: tenant_id · intent · tools_used · token_cost. This rolls up into a per-restaurant invoice line at the end of the month.

§ 14
Engineering Challenges

The hard parts.

Five problems that didn't have textbook answers — and how they were solved.

Code-switching in a single sentence

Customers write things like "اوكي keep the same و add a لارج cola". Whisper, embeddings, and the LLM all needed to handle mixed-script input without falling back to one language. Fix: embed everything with a multilingual model, normalize Arabizi to Arabic at the pre-processor, and prompt the LLM to mirror the user's register on output.

multilingual embed Arabizi normalizer register matching
LLM hallucinating prices

Early prototypes happily invented dishes and quoted plausible-but-wrong prices. Fix: three-layer defense — (1) system prompt forbids citing prices, (2) every reply runs through a price reconciler that diffs cited numbers against the cart, (3) any mismatch loops back to the planner.

prompt rule post-hoc check loop on failure
Voice notes with background noise

Egyptian street noise, kids in the background, distant TV. Out-of-the-box Whisper transcribed "كولا" as "صورة" 14% of the time. Fix: fine-tuned Whisper on a 6k-sample EG-dialect dataset, plus an "echo back what I heard" confirmation step before any order is committed.

fine-tune explicit confirm step
Latency under burst load (Ramadan iftar)

P95 spiked to 12s during peak hours. Fix: moved intent classification to Haiku (faster, cheaper, accurate enough), introduced speculative tool dispatch, and pre-warmed Pinecone connections per pod. New p95: 4.1s under 3× normal load.

model right-sizing speculative exec connection pooling
Multi-tenant prompt drift

Tenant-specific prompt overrides started growing into giant blobs that degraded model performance. Fix: structured the brand voice as a small set of slots (tone, formality, emoji usage, signature phrases) instead of free text. Compiled into the prompt at request time.

slot-based voice eval-gated changes
